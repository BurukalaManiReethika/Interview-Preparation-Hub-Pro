Hash Algorithm Reference

Hash functions are one-way — they cannot be reversed. Use SHA-256 or better for security. MD5/SHA-1 are broken for cryptographic use.

System Design Notes

Core concepts and sample designs for your system design interview.

🏛️ The SOLID Framework for System Design Interviews

ScaleEstimate load: QPS, DAU, storage needs first
ComponentsIdentify the key services and their responsibilities
Data ModelDefine entities, relationships, and storage choice
APIsDefine REST/gRPC endpoints between services
Deep DiveGo deep on one bottleneck the interviewer highlights

🔗 Design a URL Shortener (bit.ly)

Requirements: Given a long URL, return a short alias. Redirect to original on visit. 100M URLs/day write, 1B reads/day.

  • Encoding: Base62 (a-zA-Z0-9) on auto-incremented ID or MD5 first 7 chars → ~3.5 trillion combinations
  • Storage: NoSQL (Cassandra/DynamoDB) — simple key→value, write-heavy, schema-less
  • Cache: Redis for hot URLs (80/20 rule — 20% URLs get 80% traffic)
  • Redirect: 301 (permanent, cached by browser) vs 302 (temporary, always hits server for analytics)
  • Scale: CDN for redirect latency, consistent hashing for DB shards

💬 Design a Real-Time Chat (WhatsApp)

Requirements: 1-on-1 and group messaging, online presence, message delivery receipts. 50M DAU.

  • Protocol: WebSocket for persistent bidirectional connection (vs HTTP polling)
  • Message flow: Client → Chat Server → Message Queue (Kafka) → Recipient's Chat Server → Client
  • Storage: HBase / Cassandra for messages (time-series friendly, high write throughput)
  • Presence: Heartbeat every 5s → Redis TTL key. If key expires, user is offline
  • Push notifications: APNS / FCM for offline users via notification service

⏱️ Rate Limiter

Prevent abuse by limiting requests per user/IP. Common algorithms:

  • Token Bucket: Tokens added at fixed rate, consumed per request. Allows burst. Easy to implement with Redis.
  • Leaky Bucket: Requests processed at constant rate (queue). Smooths traffic, drops bursts.
  • Fixed Window: Count resets each window. Simple but edge spike problem (2× traffic at boundary).
  • Sliding Window Log: Track timestamps. Accurate but memory-heavy.
  • Sliding Window Counter: Weighted blend of prev + curr window. Good accuracy + low memory.
-- Redis token bucket (pseudocode) FUNCTION is_allowed(user_id): key = "rate:" + user_id tokens = REDIS.GET(key) or MAX_TOKENS IF tokens > 0: REDIS.DECR(key) REDIS.EXPIRE(key, WINDOW_SECONDS) RETURN true RETURN false -- rate limited

🔍 Design a Search Autocomplete

Type-ahead suggestions as the user types. Google-scale: 10M QPS, <100ms latency.

  • Trie: Stores prefixes. Each node has top-K cached completions. Query = O(prefix length).
  • Data collection: Log searches → analytics job (weekly) → update Trie → serialize to DB
  • Serving: Trie in memory on search servers. Updates via versioned snapshots, not hot-patch.
  • Scale: Shard trie by first 2 chars. CDN caches popular prefix results.

🎬 Design a Video Streaming Service (YouTube)

  • Upload: Client → chunked upload → object storage (S3). Transcoding queue (Kafka) → workers produce HLS variants (360p/720p/1080p)
  • Playback: Adaptive bitrate (ABR) — client switches quality based on bandwidth
  • CDN: Videos pushed to edge CDN nodes. Viewer routed to nearest PoP.
  • Metadata: MySQL for video info (title, owner, likes). Cassandra for comments at scale.
  • Recommendations: Offline ML pipeline → pre-computed scores → Redis → served in feed

🏗️ Key System Design Concepts Cheat Sheet

CAP TheoremCan only guarantee 2 of: Consistency, Availability, Partition Tolerance
SQL vs NoSQLSQL: ACID, joins, schema. NoSQL: scale, flexible, eventual consistency
Horizontal ScaleAdd more machines. Use consistent hashing to distribute load.
CachingCache-aside, write-through, write-back. Eviction: LRU, LFU, TTL.
Message QueueKafka/RabbitMQ decouple producers from consumers. Enables async processing.
CDNPush/pull model. Serves static assets from edge. Reduces latency globally.
Load BalancerL4 (TCP) vs L7 (HTTP). Algorithms: round-robin, least-connections, IP hash.

Coding Interview Tips

Strategies, patterns, and habits that top candidates use.

⏱️ INTERVIEW PROCESS

01

Understand Before Coding

Spend 3–5 minutes asking clarifying questions. What are the constraints? Can input be empty? Sorted? Can you modify the input? This signals maturity.

02

State Your Approach First

Before writing a single line, say your approach out loud. "I'll use a sliding window — O(n) time." Let the interviewer agree or redirect you early.

03

Start with a Brute Force

Even if obvious, mention the brute force first, then say why you can do better. Shows structured thinking and makes the optimization feel earned.

04

Test Your Code Manually

After writing, trace through a small example by hand. Fix bugs before announcing you're done. Catching your own bugs is a strong signal.

05

Analyze Time & Space

Always state Big-O for both time and space. Be specific: "O(n log n) due to sorting" beats just "O(n log n)".

06

Handle Edge Cases

Empty input, single element, duplicates, negative numbers, very large inputs, overflow. Mention them even if you don't code them all.

🧩 PROBLEM-SOLVING PATTERNS

07

Two Pointers

Use when array is sorted or you need pairs. Move pointers inward to find target sum, remove duplicates, or check palindromes. O(n) instead of O(n²).

08

Sliding Window

For subarrays/substrings of variable or fixed size. Expand right pointer, shrink left when constraint violated. Max subarray, longest substring without repeat.

09

Fast & Slow Pointers

Detect cycles (Floyd's), find middle of linked list, find cycle start. Fast moves 2 steps, slow moves 1. When they meet → cycle exists.

10

Binary Search on Answer

If you can check "is X possible?", binary search on the answer space. Useful for min/max problems: Koko Eating Bananas, Capacity to Ship Packages.

11

BFS for Shortest Path

BFS on unweighted graphs always gives shortest path. Use deque. Think in "levels" — each level is +1 step from source. Dijkstra for weighted graphs.

12

Monotonic Stack

For "next greater element", "largest rectangle in histogram". Maintain a stack that's always increasing or decreasing. Pop when invariant is violated.

13

Top-K with Heap

Min-heap of size K gives top-K largest elements in O(n log K). Faster than full sort O(n log n) when K is much smaller than n.

14

DP: Define State Clearly

dp[i] = "the answer for subproblem of size i". Write the recurrence in words before code. "dp[i] = max(dp[i-1], dp[i-2] + nums[i])".

🧠 MINDSET & HABITS

15

Think Out Loud

Interviewers can't help you if they don't know where you're stuck. Narrate your thinking even when you're uncertain. "I'm thinking a hash map here because..."

16

Use Meaningful Variable Names

In timed interviews, clarity beats brevity. Use left/right instead of l/r. prevNode instead of pn. Code you can read when stressed is code that works.

17

Practice Writing, Not Just Reading

Reading solutions gives the illusion of understanding. Solve problems from scratch on a blank file. The gap between "I understood it" and "I can write it" is where interviews are lost.

18

Learn the 15 Patterns

Sliding window, two pointers, fast-slow, intervals, cyclic sort, in-place reversal, BFS, DFS, two heaps, subsets, modified binary search, top-K, k-way merge, topological sort, DP.

19

Spaced Repetition

Review problems you've solved 1 day later, then 3 days, then 1 week. You should be able to code a problem you've seen before in under 5 minutes. If not, re-learn it.

20

Mock Interviews

At least 5 mock interviews before the real one. Pramp, interviewing.io, or a peer. The physical sensation of being watched while coding is something you must experience before it counts.
